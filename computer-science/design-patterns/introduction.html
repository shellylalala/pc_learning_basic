    #设计模式 · 介绍

    设计模式对前端的价值，不在于背出 23 种名字，而在于你能在 React 组件协作、状态流转、接口封装和复杂业务抽象里少走很多弯路。

    #一句话概况

    React 开发者学设计模式，核心不是学“经典名词”，而是建立一套处理组件协作、状态变化、行为扩展和接口隔离的稳定方法。

    #这一章讲什么

    很多教程讲设计模式，喜欢从 Java 里的类图和继承层次开始，但前端尤其是 React 开发者，真正先遇到的问题往往不是“对象怎么创建”，而是组件为什么越来越难拆、状态为什么越来越乱、相似页面为什么越写越多、一个需求改动为什么会牵一大片逻辑。

    所以这一章不会按《GoF 设计模式》的目录顺序来讲，而是按 React 开发里最常遇到的问题来组织。你会先看到那些在组件通信、状态订阅、渲染分支、请求封装、Hook 抽象里高频出现的模式，再去理解它们背后的通用思想。这样学的好处是，你学完可以立刻映射到业务代码，而不是停留在“我知道这个模式的定义，但不会用”。

    从技术栈位置上看，设计模式处在“语法会了，但工程复杂度开始上来”的那个阶段。它承接的是 JavaScript、TypeScript、React 基础，往上连接的是组件架构、状态管理、业务抽象、设计系统和工程化能力。很多人会误以为模式是“高级知识”，其实真正线上项目离不开它，因为项目一旦进入多人协作和中长期维护，代码问题通常不是功能写不出来，而是结构很快失控。

    对 React 开发者来说，最值得优先理解的是四类问题：组件之间怎么协作，状态怎么稳定流转，不同业务分支怎么优雅替换，外部系统怎么被封装进统一接口。观察者、策略、状态、适配器、外观、组合这些模式，几乎都在解决这四类问题。相反，访问者、解释器、享元这类模式不是没价值，而是更偏编辑器、低代码、编译工具链和性能极限场景，普通业务阶段没必要过早投入太多精力。

    #内容规划

    模块内容概述为什么先学React / 前端里的典型场景难度
    必须掌握：观察者模式理解订阅、通知、依赖变更怎么传播这是状态管理和组件响应更新的基础Context、Redux、Zustand、事件监听、全局通知⭐⭐
    必须掌握：策略模式把可替换行为从大量条件分支里拆出来业务分支一多，最先失控的就是 if/else表单校验、列表排序、渲染策略、请求重试策略⭐⭐
    必须掌握：状态模式用明确状态驱动 UI 和行为，而不是靠零散布尔值拼凑React 页面复杂后，最常见的问题就是状态边界混乱loading/success/error、审批流、弹窗流转、按钮禁用逻辑⭐⭐⭐
    必须掌握：装饰器 / 代理 / 外观理解增强、拦截、封装入口三种常见抽象方式这是 Hook 封装、API client、权限控制的高频手法HOC、请求拦截器、埋点包装、统一业务 SDK⭐⭐⭐
    必须掌握：适配器 / 组合 / 工厂解决接口不一致、树状结构组织、按类型生成对象组件系统和页面配置化离不开它们表单 schema、动态组件、菜单树、路由树、低代码渲染⭐⭐⭐
    值得掌握：命令 / 责任链 / 模板方法解决动作编排、流程串联、固定流程抽象当业务进入复杂交互和中后台流程时很常见撤销重做、快捷键系统、校验链、提交流程模板⭐⭐⭐
    进阶理解：建造者 / 中介者 / 桥接 / 抽象工厂 / 原型处理复杂配置、组件联动、维度解耦、对象族创建它们更偏复杂系统设计，不是日常 CRUD 第一优先级图表配置、主题系统、平台适配、组件族生成⭐⭐⭐⭐
    知道即可：访问者 / 解释器 / 享元面向 AST、DSL、规则引擎、性能复用等特殊场景知道适用问题就够了，没场景硬学收益很低Babel 插件、低代码规则、虚拟列表、图形编辑器⭐⭐⭐⭐

    #学习建议

    • 不要按“23 种模式”背诵，先按 React 里的真实问题学：组件通信先看观察者，渲染分支先看策略，状态流转先看状态模式，接口封装先看适配器和外观。
    • 初期不要沉迷类图和 UML。你更应该做的是把自己项目里一个难维护的页面拿出来，反过来问：这里是不是 if/else 太多、状态太散、接口耦合太重，然后再对应模式去重构。
    • 学模式时一定要和 Hook、组件、状态管理一起看。对前端来说，模式不是单独一门课，它本来就寄生在 useState、useReducer、自定义 Hook、API 层、组件树这些日常代码里。
    • 单例、装饰器这类模式在前端很容易被滥用。看到“全局唯一”和“包一层就能增强”时先停一下，判断它是不是会带来全局污染、调试困难和职责不清，而不是因为能用就到处用。
    • 访问者、解释器、享元这类模式现阶段知道边界就够了。除非你开始做编辑器、低代码、规则引擎、编译工具链，否则没必要在普通 React 业务阶段花大量时间深挖。

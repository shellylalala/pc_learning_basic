    #代理模式

    面向 React + TypeScript 初中级开发者：代理模式在前端最有价值的地方，不是“找个对象替另一个对象干活”，而是把权限校验、缓存、日志、节流、懒加载这些横切逻辑挡在真实业务对象前面。

    #1. 设计模式概念

    代理模式解决的是“访问某个对象之前，需要先做一层控制或增强”的问题。

    一句话理解：不让调用方直接碰真实对象，而是先经过一个代理层，由代理决定要不要放行、怎么增强、是否缓存、是否记录。

    它的核心思想不是简单地“包一层函数”，而是明确职责边界：

    • 真实对象只负责核心业务
    • 代理对象负责访问控制、附加行为和统一拦截

    代理模式解决的本质问题是：把那些和核心业务无关、但每次访问前后都要做的事情，从真实对象里拆出去。

    #2. 为什么前端需要它

    React 项目里很多逻辑，并不属于组件本身，也不属于真实业务服务本身，但它们又几乎每次访问都绕不过去。

    例如访问一个报表接口时，常常要做这些事：

    • 先检查当前用户是否有权限
    • 如果短时间内查过同一份数据，优先走缓存
    • 请求开始和结束时记录日志
    • 出错时统一返回可展示的错误信息

    如果没有代理思维，最常见的写法就是：组件里先判断权限，再查缓存，再发请求，再记日志。久而久之，调用方到处复制同样的拦截逻辑。

    React 工程里尤其容易出现这个问题，因为组件已经要处理状态、交互和渲染。如果组件再负责权限、缓存、埋点、重试、请求合并，边界会迅速变糊。

    代理模式的价值就在这里：组件只关心“我要拿数据”，至于能不能拿、需不需要缓存、要不要记日志、是不是要提示无权限，都交给代理层决定。

    #3. 真实业务场景

    场景为什么适合代理模式
    API Client / SDK 封装请求前要统一挂 token、做鉴权、做错误处理、记日志，不适合让每个页面重复写。
    图片懒加载与预加载用户真正看到图片前，先由代理对象控制加载时机和占位资源。
    权限代理某些按钮、页面、报表只允许特定角色访问，代理层适合集中控制放行逻辑。
    缓存代理相同参数的请求短时间内重复发生时，优先返回缓存，避免重复请求。
    防抖 / 限流代理搜索框、自动补全、频繁触发的点击行为，代理层适合拦截高频访问。

    这些场景说明的都是同一个问题：调用方想访问真实对象，但访问前后需要一层统一治理。

    #4. 错误示范（非常重要）

    先看一个没有代理模式的报表查询逻辑：

    async function loadReport(role: UserRole, forceRefresh: boolean) {
  if (role === "guest") {
    throw new Error("无权限查看报表");
  }

  const cacheKey = `${role}-weekly`;

  if (!forceRefresh && reportCache[cacheKey]) {
    return reportCache[cacheKey];
  }

  console.log("开始请求报表");

  const report = await fetchWeeklyReport(role);

  reportCache[cacheKey] = report;
  console.log("报表请求完成");

  return report;
}

    这段代码的问题，不是能不能工作，而是它把三种不同职责塞在一起了：

    • 权限判断
    • 缓存判断
    • 真实接口访问

    如果以后还要加请求去重、错误上报、埋点、角色降级、测试 mock，这个函数会变成“所有横切逻辑的垃圾桶”。

    更糟糕的是，组件也可能开始复制这些判断：有的地方先查权限，有的地方先读缓存，有的地方忘了记日志。最后同一个接口在不同页面上表现不一致。

    #5. 使用设计模式重构

    这次 demo 使用真实前端场景：报表服务访问代理。

    重构后的设计是：

    • ReportService 只负责“如何拿到真实报表数据”
    • ReportServiceProxy 负责权限校验、缓存和访问日志
    • React 组件只负责切换角色、触发查询、展示结果
    • 新增一种访问控制时，只需要改代理，不需要改所有调用方

    export type UserRole = "guest" | "analyst" | "admin";

export interface ReportData {
  title: string;
  revenue: number;
  conversionRate: number;
  fromCache: boolean;
}
    class ReportServiceProxy {
  async getWeeklyReport(role: UserRole, forceRefresh = false) {
    if (role === "guest") {
      throw new Error("访客没有报表查看权限");
    }

    if (!forceRefresh && this.cache.has(role)) {
      return { ...this.cache.get(role)!, fromCache: true };
    }

    return this.realService.getWeeklyReport(role);
  }
}
    const report = await reportServiceProxy.getWeeklyReport(role, forceRefresh);
setData(report);

    这里真正重要的变化是：组件不再直接接触权限判断和缓存细节，而是统一通过代理访问真实服务。

    代理模式的关键不是“多一个对象”，而是把访问控制收口到一个统一入口。

    #7. 优点

    第一，调用方更干净。React 组件只关心拿数据和渲染，不再复制权限判断、缓存命中、日志记录。

    第二，横切逻辑更集中。鉴权、缓存、日志、降级这些通用逻辑统一放在代理层，更容易维护。

    第三，扩展更平滑。以后要加请求去重、埋点、mock、限流，优先改代理，不需要满项目改调用代码。

    第四，更容易统一行为。所有页面都经过同一个代理入口，接口访问策略不会出现“这个页面记日志、那个页面不记”的分裂情况。

    第五，利于测试。真实服务和代理可以分开测试：一个测业务数据获取，一个测权限、缓存和拦截行为。

    #8. 缺点

    代理模式也不能滥用。

    如果只是一个非常简单的函数调用，没有额外访问控制需求，硬加代理只会增加间接层，让代码跳来跳去。

    另外，代理一旦承担太多职责，也会腐化：

    • 什么都想拦截
    • 什么都想缓存
    • 什么都想在代理里判断

    最后代理层会变成新的“大一统入口”。

    所以代理模式更适合访问前后确实有统一横切需求的场景，而不是为了套模式给每个服务都包一层壳。

    #9. React / 常见库中的体现（重点）

    前端工程里，代理模式几乎天天都在用，只是名字不一定叫 proxy。

    Axios 拦截器是非常典型的代理思想。组件想发请求，但真正请求前要加 token、统一处理错误、记录日志，这些事情都发生在“访问真实请求能力之前”。

    很多团队封装 API Client 或 BFF SDK 时，也是在做代理：页面并不直接碰底层 fetch，而是先经过一层带权限、缓存、降级和监控的访问层。

    图片懒加载组件也很像代理模式。你以为自己在直接显示真实图片，其实中间有一个代理层在决定何时开始加载、何时展示占位图。

    React Query 虽然核心更偏缓存与服务状态管理，但它也体现了代理思想：组件不是直接请求接口，而是通过统一查询层读取缓存、触发请求、接收错误和状态。

    Next.js 中常见的数据获取封装、中间层鉴权、边缘缓存，也都可以从代理模式角度理解：真实资源访问前，先经过一层控制入口。

    从 Hooks 设计角度看，自定义 Hook 也经常承担“代理”角色。组件调用 useReport()，但 Hook 内部可能已经帮你做了缓存、权限处理和统一错误提示。

    #10. 总结

    代理模式最核心的一句话是：不要让调用方直接接触真实对象，而是在它前面放一个统一的访问控制层。

    真正应该学会的，不是背“代理和装饰器有什么区别”，而是当你发现权限、缓存、日志、限流这些逻辑开始在多个页面里重复出现时，能意识到：这里需要一个代理入口来收口横切逻辑。

    对 React 开发者来说，代理模式最大的意义，就是把访问控制从组件里搬出去，让组件重新回到状态、交互和渲染本身。
